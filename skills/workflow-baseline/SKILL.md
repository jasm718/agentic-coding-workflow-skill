# Workflow Baseline

逐项建立项目需求基线，并把确认后的结果写入 `docs/project.md`。

## Workflow

1. 先检查仓库根目录下是否存在 `docs/project.md`。
2. 如果文件不存在，创建它；如果存在，先读取当前内容，避免覆盖用户手改内容。
3. 严格按以下顺序推进：
   - `项目目标`
   - `非目标`
   - `技术约束`
   - `验收标准`
4. 一次只讨论一个主题，不要并行讨论多个主题。
5. 对当前主题执行固定流程：
   - 提一个问题，聚焦当前主题
   - 等用户回答
   - 把答案整理成适合写入文档的简洁表述
   - 请求用户确认
   - 只有用户明确回复 `确认` 后，才写入 `docs/project.md`
6. 写入时只更新当前主题对应章节，不追加重复标题，不覆盖其他章节。
7. 当前主题确认并写入后，再进入下一个主题。
8. 全部主题完成后，提醒用户审阅 `docs/project.md`，如果要继续下一阶段，回复 `继续`。

## Document Rules

- 始终把 `docs/project.md` 视为该阶段唯一主文档。
- 使用固定二级标题：
  - `## 项目目标`
  - `## 非目标`
  - `## 技术约束`
  - `## 验收标准`
- 如果章节已存在，用最新确认内容整体替换该章节正文，不做兼容性合并。
- 在用户未回复 `确认` 之前，禁止写文件。

## Fail Fast

- 如果无法创建或写入 `docs/project.md`，立即报错并停止。
- 如果用户回答明显不完整或存在歧义，立即追问，不自行补全。
- 如果用户试图跳到下一阶段，先明确指出当前基线尚未完成。
- 不要顺手讨论页面结构、数据模型、API 设计或实现细节。
