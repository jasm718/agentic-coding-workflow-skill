# 收束一句话需求到项目基线

通过自然的协作式对话，将想法逐步转化为完整的设计和规格说明。

从理解当前项目背景及项目上下文，然后通过一次只提出一个方面的不超过4个问题的方式不断细化想法。一旦你清楚了某一方面的内容，就提出最终结果并获得用户的确认，并且写入对应文档章节。

不要调用任何实现相关技能、编写任何代码、搭建项目结构或进行任何实现操作。无论项目看起来多简单，这一原则都必须始终遵守。

## process

**检查现状：**

- 检查当前项目状态（文件、文档、最近提交）

**确定范围：**

- 在提出详细问题之前先评估范围：如果请求描述的是多个彼此独立的子系统（例如“做一个包含聊天、文件存储、计费和分析的平台”），要立刻指出这一点。不要把本来就需要先拆分的项目，直接拿来细化细节。
- 如果项目太大，无法用一份规格涵盖，就帮助用户把它拆成多个子项目：哪些部分彼此独立，它们之间怎么关联，应该按什么顺序做。然后让用户选择一个子项目继续流程。

**头脑风暴：**

- 对于范围合适的项目，一轮只问一个方面、不超过3个的问题来逐步细化`项目目标`，`技术约束`，`成功标准`等其他你觉得必要的内容，每个方面你如果觉得没有足够了解，就继续追问，直到你能够总结出一个清晰的、完整的、可执行的结果为止。
- 尽可能优先使用选择题并给出a,b,c等选项，但开放式问题也可以，永远留给用户一个选项"其他"来提供不在选项中的答案。
- 一旦你认为自己已经理解要构建什么了，就结束提问，进入下一个阶段。

**落成文档：**

- 将你的理解严格按照`项目目标`，`技术约束`，`成功标准`以及你觉得必要的章节展示出来，并获得用户确认。
- 用户确认后，以精简的，可读性好的文字写入`docs/project.md`。

## Rules

- 始终把 `docs/project.md` 视为该阶段唯一主文档。
- 给出 2-3 种方案，说明取舍，并给出你的推荐，且加粗显示推荐项。
- 如果章节已存在，用最新确认内容整体替换该章节正文，不做兼容性合并。
- 在用户未回复 `确认` 之前，禁止写文件。

## Fail Fast

- 如果无法创建或写入 `docs/project.md`，立即告知用户问题并停止。
- 如果用户回答明显不完整或存在歧义，立即追问并附带自己的建议和想法，不自行补全。
- 如果用户试图跳到下一阶段，先明确指出当前基线尚未完成。
- 不要顺手讨论其他内容。
