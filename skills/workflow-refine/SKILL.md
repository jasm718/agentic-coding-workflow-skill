# Workflow Refine

基于 `docs/project.md` 继续逐项收敛需求，并把确认后的结果写回同一份文档。

## Workflow

1. 先读取 `docs/project.md`。
2. 如果文件不存在，立即报错并停止，不要假设基线已经完成。
3. 按以下候选顺序推进：
   - `页面结构`
   - `组件拆分`
   - `数据模型`
   - `API设计`
   - `目录结构`
   - `日志点`
   - `异常处理`
   - `低保真原型说明`
4. 对每个候选主题先判断是否适用当前项目。
5. 如果不适用，明确说明该主题被跳过，然后继续下一个主题。
6. 如果适用，对当前主题执行固定流程：
   - 提一个问题，聚焦当前主题
   - 等用户回答
   - 把答案整理成适合写入文档的简洁表述
   - 请求用户确认
   - 只有用户明确回复 `确认` 后，才写回 `docs/project.md`
7. 一次只讨论一个主题，不要把多个主题合并到同一轮消息中。
8. 所有需要的主题完成后，提醒用户审阅 `docs/project.md`，如果要开始拆任务，回复 `继续`。

## Document Rules

- 始终先重新读取 `docs/project.md`，再开始新的讨论轮次。
- 使用固定二级标题，并且只保留实际需要的章节。
- 如果章节已存在，用最新确认内容整体替换该章节正文，不做合并。
- 如果章节不存在，按固定标题顺序插入。
- 在用户未回复 `确认` 之前，禁止写文件。

## Fail Fast

- 如果 `docs/project.md` 缺失，立即停止。
- 如果当前主题是否适用无法判断，直接提问，不要默认需要或不需要。
- 如果用户要求同时讨论多个主题，主动收窄到当前主题。
- 不要进入任务拆解或代码实现阶段。
