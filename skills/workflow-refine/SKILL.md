# 细化需求

基于 `docs/project.md` 继续逐项收敛需求，并把确认后的结果写回同一份文档。

## Workflow

**了解现状：**

- 先读取 `docs/project.md`，如果文件不存在，立即报错并停止，不要假设基线已经完成。

**逐步推进：**

- 对以下每个主题先判断是否适用当前项目，如果不适用跳过该主题，如果适用则参照`@workflow-baseline`中对`目标`，`范围`，`技术约束`，`交付标准`的讨论方式进行讨论：
  - `组件拆分`
  - `数据模型`
  - `API设计`
  - `目录结构`
  - `日志点`
  - `异常处理`
  - `低保真原型`

**优化输出：**

- 根据你对刚才讨论内容的理解，对信息进行如下修改（没有则无需修改），并展示出来让用户确认：
  - 优化`组件拆分`部分：使用ascii图格式画成组件关系图，展示组件之间的调用关系和数据流向。
  - 优化`数据模型`部分：将数据模型设计成表格形式，包含字段、类型、说明信息。
  - 优化`API设计`部分：画成表格形式，包含接口名称、请求方法、路径、参数、返回值说明等信息。
  - 优化`目录结构`部分：使用ascii目录树格式展示项目目录结构，并标明各重要文件/目录的职责。
  - 优化`低保真原型`部分：使用ascii图像向用户展示界面布局和交互流程。
- 用户确认后，写入`docs/project.md`。

## Rules

- 始终先重新读取 `docs/project.md`，再开始讨论。
- 在用户未回复 `确认` 之前，禁止写文件。
- 如果 `docs/project.md` 缺失，立即停止。
- 如果当前主题是否适用无法判断，直接提问，不要默认需要或不需要。
- 如果用户要求同时讨论多个主题，主动收窄到当前主题。
- 不要进入任务拆解或代码实现阶段。
